# Implementation plan: admin-dashboard

**Spec:** [docs/specs/admin-dashboard.md](../../docs/specs/admin-dashboard.md)
**Spec dependencies read:** auth, bookings, messaging, payments, reporting (operator-facing)
**Playbooks read:** security.md, gdpr.md

---

## Pre-flight flags (read these first)

- **PCI scope:** unchanged. Dashboard reads `payments` metadata only (Stripe intent IDs, status, amount in pence). No PAN / CVV / card data ever flows through the admin path. SAQ-A unaffected.
- **Sub-processors:** no new sub-processor. Stripe is already in [docs/playbooks/gdpr.md](../../docs/playbooks/gdpr.md) §sub-processors; the live MRR pull is additional API calls to an existing relationship, not a new one.
- **PII handling:** the admin dashboard MUST NOT decrypt guest PII. Every metric is an aggregate (count, sum, status histogram) over indexed columns. Drill-down surfaces operator emails (`users.email`, already plaintext for login) but never decrypts `guests.*_cipher`. This is enforced as an integration-test invariant in PR2 (see below). No `lib/security/crypto.ts` calls from `lib/server/admin/dashboard/**`.
- **RLS:** **no new org-scoped tables**, so no RLS-on-org-data work. One new platform-only table (`platform_audit_log`) ships RLS-enabled with a deny-all policy — service-role bypass via `adminDb()` is the only access path.

---

## Spec-correction note (small, before code)

The spec at [docs/specs/admin-dashboard.md](../../docs/specs/admin-dashboard.md) names paths as `lib/admin/...`. The actual location should be **`lib/server/admin/dashboard/...`** because [lib/server/admin/db.ts](../../lib/server/admin/db.ts) is import-allowlisted by `code-reviewer` (per [docs/playbooks/security.md](../../docs/playbooks/security.md) §Cross-tenant bugs). Putting metric query functions outside that directory would either (a) break the allowlist invariant, or (b) require a whitelist amendment. Cleaner to nest under the existing dir.

PR1 includes a one-line spec edit to update the path references. No functional impact.

## 3. Migrations

**One migration in PR1.** No further schema changes in PR2/3/4.

`drizzle/migrations/0020_<auto>.sql` (generated by `pnpm db:generate` after editing `lib/db/schema.ts`):

```sql
-- Platform-staff audit log. Distinct from public.audit_log (which is
-- org-scoped). This table records cross-org actions taken by Tablekit
-- staff via the admin dashboard. NOT org-scoped.
--
-- Access: writes ONLY via lib/server/admin/dashboard/audit.ts (which
-- uses adminDb() / service_role, bypassing RLS). Operators must never
-- read this table — RLS enforces deny-all to authenticated/anon.

CREATE TABLE "platform_audit_log" (
  "id"          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  "actor_email" citext NOT NULL,
  "action"      text NOT NULL,
  "target_type" text,
  "target_id"   uuid,
  "metadata"    jsonb NOT NULL DEFAULT '{}'::jsonb,
  "created_at"  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX "platform_audit_log_created_at_idx"
  ON "platform_audit_log" ("created_at" DESC);

ALTER TABLE "platform_audit_log" ENABLE ROW LEVEL SECURITY;
```

(RLS policy SQL is in §4 below.)

No data backfill. No column changes to existing tables. Forward-only — Drizzle convention.

## 4. RLS policies

### `platform_audit_log`

Not org-scoped. Deny-all to `authenticated` and `anon`; service_role bypass via Postgres `BYPASSRLS` is the only access path. Satisfies `scripts/check-rls.ts` (RLS enabled + ≥1 policy present).

```sql
CREATE POLICY "platform_audit_log_no_access"
  ON "platform_audit_log"
  FOR ALL
  TO authenticated, anon
  USING (false)
  WITH CHECK (false);
```

### Existing tables

**No changes.** All admin queries go through `adminDb()` (service_role, RLS bypass). Per [docs/playbooks/security.md:83](../../docs/playbooks/security.md#L83), use of `service_role` outside `lib/server/admin/**` is flagged by `code-reviewer` — this plan keeps every admin query inside `lib/server/admin/dashboard/**`, preserving the invariant.

## 6. Risks (review-time watch list)

| # | Risk | Mitigation |
|---|---|---|
| R1 | **`adminDb()` RLS bypass leaks guest PII into the dashboard.** Cross-org by design, but if a query joins through a guest field and decrypts, it leaks. | All metrics are aggregates on counts/timestamps/statuses. Test `admin-metrics.test.ts` spies on `lib/security/crypto.ts` and fails if called from any admin code path. Code-reviewer should reject any admin file that imports `crypto`. |
| R2 | **Stripe customer ID semantics on `organisations`.** Column exists but its current usage may be guest-payments customer (Connect), not billing customer. Per-org MRR contribution would be wrong if we read it as billing. | v1 MRR is a *platform total*, not per-org. Per-org MRR contribution is **explicitly deferred** until either (a) we confirm the column's intent by reading current writers, or (b) we add `stripe_billing_customer_id`. Document as a v1 limitation in the financials page UI. |
| R3 | **`ADMIN_EMAILS` env var misconfigured.** Empty in prod = locks Ben out. Wildcard / extra emails = opens the door. | Middleware logs a console warning at startup if `ADMIN_EMAILS` is empty in `NODE_ENV=production`. `requirePlatformAdmin()` parses the env on every call (no module-init capture) so a hot-reload/redeploy fixes typos immediately. Test parses `"a@b.co, c@d.co"` correctly. |
| R4 | **Middleware regression affecting all routes.** Next.js proxy is global by default. | `export const config = { matcher: ['/admin/:path*'] }` — operator/widget/marketing routes never see this proxy. Verified by reading the matcher in code-review and by the e2e suite (operator pages still load). |
| R5 | **Stripe API outage blocks the financials page.** | `unstable_cache` with 5-min revalidate, plus `try/catch` around the live call returning `{ data: lastCached, degraded: true, asOf }`. Page shows a yellow banner; never throws. Unit-tested. |
| R6 | **`platform_audit_log` schema drift.** A future migration adds an org-scoping column "for analytics" and accidentally exposes via RLS. | Comment in schema.ts marks the table as `// PLATFORM-ONLY — never add organisation_id`. RLS deny-all policy + `gdpr-auditor` review on any future change. |
| R7 | **Bundle bloat from recharts on operator pages.** | Dynamic-import recharts inside a client component (`components/admin/sparkline.tsx`); only loaded under `app/(admin)/`. Verified in PR3 by checking the operator-page Network tab in DevTools (acceptance criterion in spec). |
| R8 | **Cross-tenant write via admin UI.** v1 has no write surfaces — all read. Future PRs adding writes (impersonation, manual subscription edits) re-enter this risk surface. | Spec §Non-goals explicitly excludes writes from v1. Any future write needs its own RLS / authz design + `code-reviewer` + `security-reviewer` pass. |

## 7. Rollback plan

Per PR:

- **PR1** — `git revert <merge>`. The migration `0020_*` is forward-only per repo convention. The `platform_audit_log` table can stay (it's empty if PR1 is the only thing rolled back, and harmless). If we want to drop it entirely, follow up with a `0021_drop_platform_audit_log.sql` — but that's optional, not required for rollback.
- **PR2 / PR3 / PR4** — pure code reverts; no schema. `git revert <merge>` and redeploy.

**Kill switch (immediate, no deploy):** clear `ADMIN_EMAILS` from Vercel env and redeploy → proxy fails closed for everyone, including Ben. Page returns 404.

**Live Stripe outage:** no rollback needed — the degraded path keeps the page rendering.
